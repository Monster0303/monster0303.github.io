一、软件的架构模式¶

软件从设计上，有以下几种架构：

  • 分层架构：把要实现的功能在以下四个层级中抽象的表现出来，各层级之间是隔离的，通过调用接口来完成工作；
    • 表现层：主要处理业务逻辑本身；
    • 业务层
    • 持久层
    • 数据库层：通过调用数据库，把持久层的数据保存在对应的数据库中；
  • 事件驱动架构模式：可以理解是一个分布式异步架构模式，模块之间的调度高度解耦；
  • 微内核架构：即插件式架构，程序核心本身设计的非常小，各种其他功能通过扩展模块的方式插入到核心中，比如 Nginx 就是这样的架构；
  • 微服务架构：把整体系统功能的每一项功能，都独立为一个又一个的发布单元，每个单元都是一个独立的服务提供者，各服务之间通过服务治理系统来完成服务组件之间的交互和通信。
    • API REST-based
    • application REST-based
    • 中心化消息
  • 基于空间的架构：云原生的应用程序。

二、Tomcat 是什么¶

了解 Tomcat 之前先了解一下 Java，下图是 Java 的技术体系：

Java 技术体系¶

从下网上看，可以看出有以下几个部分：

  • JVM：是 Java 虚拟机，一个专门运行 Java 的环境；
  • JRE：是 JVM + 各种 JAVA2 SE API 的 标准类库 接口；
  • JDK：是 JRE + Java 的开发工具，开源社区版叫 OpenJDK。

这其中，JDK 包含了 JRE，JRE 又包含了 JVM。

Java 的类库接口分为：

  • 标准版：JAVA2 SE(Standard Edition) API；
  • 企业版：JAVA2 EE(Enterprise Edition) API。

其中 JRE 中包含的类库就是 JAVA2 SE 标准类库，而企业版则是 JAVA2 EE 企业类库，企业版比标准版多出很多的库接口，如 Applet 和 jsp 等等。

Tomcat¶

而 Tomcat 就是在 JRE 的基础上，又包含了 Applet 和 jsp 这两个最常用的企业类库，可以看做是 J2EE 的不完整实现。

Tomcat 是 Apache 的 ASF 旗下已经毕业的开源项目，本身其实也是用 Java 程序编写的，是一个专门为了运行 jsp(java server page) 的容器，而 Tomcat 的代码名则叫 catalina。

实现了此类功能的软件其实有很多，比如：

  • 商业实现：
    • WebSphere
    • WebLogic
    • Oc4j
    • Glassfish
    • Geronimo
    • JOnAS
    • JBoss
  • 开源实现：
    • Tomcat
    • Jetty
    • Resin

JSP 代码的运行流程¶

JSP 其实也是一种 Java 代码，所以先了解一下 Java 代码的运行流程。

Java 代码的运行流程：

  • .java 代码会被 java C 编译为 .class (servlet 程序)，然后 在 JVM 中运行；
  • JVM 则负载加载程序的类文件 *.class，及程序的类文件依赖到的其它的类文件而后运行；
  • 运行时整个表现为一个 JVM 进程。

JSP 代码的运行流程：

  • JSP 的代码运行第一步是通过组件 jsp parser 把 *.jsp 转换为 *.java；
  • 然后就和 Java 程序运行时一样，.java 代码被 java C 编译为 .class (servlet 程序)，然后 在 JVM 中运行。


而编译好的 .java 和 *.class 文件会缓存在 /tomcat/work 目录下，加速下次使用，所以程序第一次部署后应该预热一下 Tomcat。

Tomcat 本身也是个 Java 程序，所以要运行 Tomcat 就得先安装好 Java 的运行环境。

安装 Tomcat¶

接下来就可以安装 Tomcat 了。

Tomcat 也有两种安装方法，可以直接使用 yum 从 Base 仓库安装，也可以从官网下载编译好的 tar 包解压安装。

1. 通过 yum 安装¶

可以直接使用 yum 安装，Base仓库是 7.x 的版本。

  • rpm 包安装的程序环境：
    • 配置文件目录：/etc/tomcat；
    • 主配置文件：server.xml；
  • 网页文件的存放位置：/var/lib/tomcat/webapps/
    • examples
    • manager
    • host-manager
    • docs
  • Unit File：tomcat.service；
  • 环境配置文件：/etc/sysconfig/tomcat。

通过 systemd 就能直接启动 Tomcat 了。

如果想用新版，需要从官网下载，当前版本是 8.5.43；

下载的是编译好的 Java 二进制程序，接下来展开压缩包到指定位置即可完成安装。

Tomcat 的目录结构¶

  • /bin/：脚本，及启动时用到的类；
  • /conf/：配置文件目录；
  • /lib/：库文件，Java 类库，jar；
  • /logs/：日志文件目录；
  • /temp/：临时文件目录；
  • /webapps/：网页文件的默认目录；
  • /work/：工作目录；

配置 Tomcat 程序的权限¶

注意：建议使用普通用户运行 Tomcat，而不是 root，所以先创建普通账户并修改权限。

但是对 logs, temp, work 这些目录应该有写权限，而且对 conf/* 应该有读的权限，否则会因为加载不了配置信息而无法启动。

四、Tomcat 的程序结构¶

配置文件介绍¶

Tomcat 的配置文件放置在 tomcat/conf/ 下，yum 安装的话会在 /etc/tomcat/conf/下。

  • server.xml：主配置文件；
  • web.xml：提供默认的与部署相关的配置；每个 webapp 都可以有自己的 web.xml，存放位置为 WEB-INF/ 目录中，如果没有，则此文件会提供默认部署配置；
  • context.xml：提供默认的配置文件；每个 webapp 都可以有专用的配置文件，它通常由专用的配置文件 context.xml 来定义，其存放位置为 WEB-INF/ 目录中；如果没有，此文件会提供默认配置；
  • tomcat-users.xml：关于 basic 用户认证的账号和密码文件；
  • catalina.policy：当使用 -security 选项启动 Tomcat 时，用于为 Tomcat 设置安全策略；
  • catalina.properties：主要配置 Tomcat 的安全设置、类加载设置、不需要扫描的类设置、字符缓存设置，以及一些与 JVM 调优相关参数；
  • /tomcat.conf：一些变量的配置；
  • logging.properties：日志系统相关的配置。

JSP WebAPP 的组织结构¶

  • tomcat/weball/：webapps 的根目录，放置各种与站点相关的目录；
    • index.jsp, index.html：主页文件；
    • WEB-INF/：当前 webapp 的私有资源路径；通常用于存储当前 webapp 的 web.xml 和 context.xml 配置文件；
    • META-INF/：类似于 WEB-INF/；
    • classes/：类文件，当前 webapp 所提供的类；
    • lib/：类文件，当前 webapp 所提供的被打包为 jar 格式的类；
    • ROOT：存放网站首页的目录。

webapp 归档格式¶

webapp 在发布时通常是被打包为一个归档文件，有如下几种格式：

  • *.war：webapp；
  • *.jar：EJB 的类打包文件；
  • *.rar：资源适配器类打包文件；
  • *.ear：企业级 webapp。

五、JSP 部署 webapp 的相关操作¶

部署操作有以下几种¶

  • 部署(deploy)流程:
    1. 将 webapp 的源文件放置于目标目录(网页程序文件存放目录)，配置 Tomcat 服务器能够基于 web.xml 和 context.xml 文件中定义的路径来访问此 webapp；
    2. 将其特有的类和依赖的类通过 class loader 装载至 JVM 虚拟机；
  • 部署有两种方式：
    1. 自动部署：由 Tomcat 自动完成；
    2. 手动部署：
      (1) 冷部署：把 webapp 复制到指定的位置，而后启动 Tomcat；
      (2) 热部署：在不停止 Tomcat 的前提下进行部署；部署工具包括：manager、ant 脚本、tcd(tomcat client deployer) 等；
  • 反部署(undeploy)：
    • 停止 webapp，并从 Tomcat 实例上卸载 webapp；
  • start：启动处于停止状态的 webapp；
  • stop：停止 webapp，不再向用户提供服务；其类依然在 jvm 上；
  • redeploy：重新部署；

六、部署 Tomcat 的两个管理应用¶

  1. Manager：管理 webapps 应用程序；
  2. Host Manager：管理虚拟主机。

登录管理界面需要先创建账户，并为每个账户分配不同的角色，每个角色的权限也各不相同

    • manager-script：允许访问文本界面和状态页；
    • manager-jmx：允许访问 JMX 代理和状态页；
    • manager-status：只允许访问状态页；
  • Host Manager 的角色：
    • admin-gui：Host Manager 的管理角色。

七、server.xml 中常用组件的配置¶

server.xml 是 Tomcat 的核心配置文件，内部通过组件来定义，每一个组件都由一个 Java 的“类”实现，这些组件大体可分为以下几个类型：

  • 顶级组件：Server；
  • 服务类组件：Service；
  • 连接器组件：http, https, ajp(apache jserv protocol)；
  • 容器类：Engine, Host, Context；
  • 被嵌套类：valve, logger, realm, loader, manager, …；
  • 集群类组件：listener, cluster, …；

组件的框架¶

注意：server.xml 不应该让其他用户有权限访问。

下面了解各组件的功能和使用方法。

Server 组件¶

默认配置：<Server port="8005" shutdown="SHUTDOWN">。

Server 代表了 tomcat instance，表现为一个 Java 进程，监听在 本机地址 的 8005 端口，属于管理端口，只接收 SHUTDOWN 关键字；

  • 各 Server 监听的端口不能相同，因此，在同一物理主机启动多个实例时，需要修改其监听端口为不同的端口；

注意：

  • 因为可使用 SHUTDOWN 关键字直接关闭 Tomcat 的 Java 进程。
  • 这个操作非常危险，应关闭此端口或更换为复杂的关键字。

替换关键词

Service 组件¶

用于实现将一个或多个 connector 组件关联至一个 engine 组件。

Connector 组件¶

定义属于整个 Tomcat 的监听地址和端口，负责接收请求。

  • 常见的连接器有三类：
    • http
    • https
    • ajp

进入 Tomcat 的请求可分为两类：

  1. standalone: 直接接收来自于客户端浏览器的请求；

  2. 由其它的 web server 反代，请求是来自前端的反代服务器；

    • nginx -> http connector -> tomcat；
    • httpd(proxy_http_module) -> http connector -> tomcat；
    • httpd(proxy_ajp_module) -> ajp connector -> tomcat；
    • httpd(mod_jk) -> ajp connector -> tomcat。

连接器可以使用的一些属性：

  • port="8080"：定义端口。注意：Tomcat 最好使用普通用户启动，所以无法使用 1024 以下的端口；
  • protocol="HTTP/1.1"：定义默认使用的协议版本；
  • connectionTimeout="20000"：连接超时时长，单位毫秒；
  • address：监听的 IP 地址，默认为本机所有可用地址；
  • maxThreads：最大并发连接数，默认为 200；
  • enableLookups：是否启用DNS查询功能；
  • URIEncoding="UTF-8"：如果 uri 内有中文，一定要指定编码格式为 UTF8；
  • acceptCount：等待队列的最大长度；
  • secure：启用 SSL；
  • sslProtocol：定义 SSL 证书、版本、私有文件。

Engine 组件¶

每个 Engine 代表一个 Servlet 实例，即 servlet 引擎，是可以运行代码的位置，其内部可以一个或多个 host 组件来定义虚拟主机，有多个虚拟主机时通常需要通过 defaultHost 属性来定义默认的虚拟主机；

常用属性：

  • name="Catalina"：基本上均为 Catalina；
  • defaultHost="localhost"：定义默认的虚拟主机，使用 Host 中定义的名字；
  • jvmRoute=TAG：基于 cookie 的会话绑定时，赋予当前 Engine 一个唯一的标识符。

Host 组件¶

Host 其实就是定义基于主机名的虚拟主机，位于 Engine 内部用于接收请求并进行相应处理。

注意：Tomcat 仅支持基于主机名的虚拟主机。

Host 基础配置¶

常用属性说明¶

  • name：定义虚拟主机名；
  • appBase：定义网页文件根目录，指存放非归档的 web 应用程序或归档的 WAR 文件目录路径；
    • 注意：使用相对路径时是基于 $CATALINA_BASE 变量所定义的路径；
  • unpackWARs：如果 appBase 内有一个 WAR 文件，是否自动展开；
  • autoDeploy：在 Tomcat 处于运行状态时，有新的某 webapp 放置于 appBase 所定义的目录中时，是否自动将其部署至 Tomcat。
    • 生产环境不建议开启。

Context 组件¶

可以自定义 url 和网页文件目录的对应关系，类似于 Apache 中的 alias 别名。

Context 基础配置¶

  • Context path="/PATH"：定义 uri 路径；
  • docBase="/PATH/TO/SOMEDIR"：网页文件路径；
  • reloadable=""：是否支持自动重载。

Valve 组件¶

使用 Valve 在 Host 中定义一种功能阀门。

定义访问日志的 Valve¶

使用 org.apache.catalina.valves.AccessLogValve

  • directory="logs"：日志存放目录，相对路径为 $CATALINA_BASE 的目录下；
  • prefix="localhost_access_log"：日志文件的前缀，中间是日期；
  • suffix=".txt"：日志文件的后缀；
  • pattern="%h %l %u %t &quot;%r&quot; %s %b"：定义日志格式。
