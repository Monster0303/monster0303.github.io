实战详解 - 使用 Nginx 负载均衡 Docker 中的 WordPress 和 PhpMyAdmin

一、实验介绍¶

使用物理机作为 Nginx，全权代理后端运行着 WordPress 的两台 httpd 容器，并对请求进行负载均衡。

为了保证数据的一致性，两台 httpd 连接同一台 mysql 数据库，并且容器的生产数据通过 Docker 存储卷映射到物理机。

为了提高性能，最后会在 Nginx 中启动缓存。

拓扑图如下：

下面开始演示部署过程：

二、环境准备¶

1、在物理机创建生产数据的 Docker 存储卷目录。

2、因为 Docker 默认的 bridge 网络无法为容器指定 IP，容器重启后 IP 会变动，这很尴尬，所以这里我们自己创建一个 Docker 网络 wordpress。

三、准备数据库¶

1、创建并启动 mariadb 的容器。

2、创建并启动 phpmyadmin 的容器，为了前期部署方便，发布到宿主机的 8080 端口。

3、客户端浏览器打开 192.168.50.11:8080，使用 phpmyadmin 来创建 WordPress 的数据库。
用户名密码使用 mysql 的 ROOT 账户。

4、为 WordPress 创建账户并创建同名数据库，授予权限。

到此为止数据库就准备好了

四、部署 WordPress¶

1、从官网下载站点文件，并解压缩到 物理机的存储卷目录。

2、从模板创建 WordPress 的配置文件。

  • 编辑数据库信息。

3、使用 Dockerfile 创建 httpd 镜像。

注意：centos 自带 php 版本太老，需要使用 REMI 源来安装 php7.3 版，并安装连接数据库的组件 php73-php-mysqlnd。

Dockerfile 内容：vim /data/httpd/Dockerfile

4、构建镜像。

5、启动 2 个 httpd。

五、部署 Nginx 代理¶

1、安装 Nginx

2、注释掉 /etc/nginx/nginx.conf 中默认的 server {} 配置段，避免冲突。

3、编辑代理配置文件 vim /etc/nginx/conf.d/wordpress.conf

注意：

  • 必须通过设置 proxy_set_header 把真实客户端的报文首部信息传给后端 httpd，不然 CSS 样式无法显示。
  • nginx.conf 默认配置下，会把 /etc/nginx/conf.d 目录下的所以配置文件 include 进 http {}，所以 upstream 前不应该再加 http {}。

4、启动 Nginx

六、配置 WordPress¶

访问 http://192.168.50.11 填写站点信息后，点击安装。
WordPress 配置就完成了，非常简单！

可以登录博客管理后台，我改了个主题，再次访问 http://192.168.50.11，就看到了。

默认情况下 httpd 的 access.log 日志里，客户端的 IP 是代理服务器的。
不过我们之前在 Nginx 中设置了把真实客户端的 IP 通过 X-Real-IP 传给 httpd_1 和 httpd_2，那么在 httpd 中修改一下 httpd 的日志格式就能看到真实客户端 IP 了。

最后别忘了把 Nginx 负载均衡配置中的 ip_hash 去掉，就变成默认的轮询负载了。

刷新 8 次页面，查看两个 httpd 的 access.log 日志，可以从访问时间看出，是轮询负载的。
并且能看到客户端的真实 IP 为 192.168.50.3。

到此，Nginx 负载均衡 Docker 容器内的 WordPress 就完成了！

七、启动 Nginx 缓存¶

为了提高性能，可以打开 Nginx 缓存。

先用 AB 做压力测试，看看不打开缓存时的性能。

可以看到无比异常特别的差！平均每秒只能打开页面 20 个左右。

配置打开 Nginx 缓存¶

在之前的基础上，添加缓存的配置。

vim /etc/nginx/conf/wordpress.conf

再次使用 AB 测试压力。

可以看到平均每秒打开的页面提高到了 7000 多个，性能提升非常好！是不是感觉心情也变好了呢？
