一、Nginx 反向代理¶

在代理中有两种模式，分为 正向代理 和 反向代理。

  • 正向代理
    • 代表请求方，去访问服务方。举个简单易懂的例子 --> SSR
    • SNAT
  • 反向代理
    • 代表服务方，对外提供服务，别人来访问你。
    • DNAT

而 Nginx 里最核心的功能就是反向代理，和 LVS 不一样的是，Nginx 是工作在七层的软件，能理解报文中更多的信息，进行更高级的代理方式，支持的代理内容有：

  • http
  • mail
  • stream

二、HTTP 代理¶

ngx_http_proxy_module 反代模块¶

Nginx 是高度模块化的，使用这个模块来开启反代功能。

proxy_pass¶

  • 反代的核心配置，设置将请求代理到哪一台服务器
  • 语法：proxy_pass URL;
  • 可用位置：location, if, limit_except

需要特别注意：
当 proxy_pass 后面的路径不带 uri 时，其会将 location 的 uri 传递给后端主机。

当 proxy_pass 后面的路径是一个 uri 时，proxy_pass 的 uri 会把 location 的 uri 顶替掉。

如果 location 定义其 uri 时使用了正则表达式的模式，或在 if 语句或 limt_execept 中使用 proxy_pass 指令，则 proxy_pass 之后必须不能使用 uri。用户请求时传递的 uri 将直接附加在代理到的服务的之后。

proxy_set_header¶

  • 设定发往后端主机的请求报文的请求首部的值。
  • proxy_set_header FIELD VALUE;
  • 可用位置：http, server, location

后端 http 配置文件中添加一个日志字段，被代理的服务器就能看到发请求报文的客户端 IP 了。

proxy_hide_header¶

  • 语法：proxy_hide_header field;
  • Nginx 默认情况下会隐藏主机信息，包括 “Date”, “Server”, “X-Pad”, and “X-Accel-...” 等，这里可以设置额外的响应头，不发给客户端。
  • 相反的，如果希望允许传递某些响应头给客户端，可以使用 proxy_pass_header 指令。

proxy_connect_timeout¶

  • 代理服务器与后端服务器建立连接的超时时间
  • proxy_connect_timeout 60;
  • 默认为 60s，不应该超过 75s。

proxy_read_timeout¶

  • 从后端接收响应报文的超时时长。
    proxy_read_timeout 60;
  • 默认为 60s。

proxy_send_timeout¶

  • 向后端发送请求报文的超时时长
    proxy_send_timeout 60;
  • 默认为 60s。

ngx_http_headers_module 自定义头模块¶

proxy_set_header 和 proxy_hide_header 只能在代理模式下操纵请求和响应报文的首部，而这个模块不仅可以在代理模式时使用，也能在非代理时使用，将响应给客户端的响应报文中添加自定义首部，或修改指定首部的值。

  • 可用位置：http, server, location, if in location

add_header¶

  • 添加自定义首部。
  • add_header name value [always];

示例：

  • add_header X-Via $server_addr;
  • add_header X-Accel $server_name;

expires¶

  • expires [modified] time;
  • 用于定义 Expire 或 Cache-Control 首部的值多久过期。

示例：

  • expires epoch | max | off;

三、HTTP 缓存¶

Nginx 作为反向代理服务器时，通过将经常读取的文件保存一份在本地服务器上的方式，提高代理速度。
因为所有的数据都有 局部性，会产生热区效应，使用缓存机制提高性能。

局部性：

  • 时间局部性：如果一个数据最近被访问过，那么很快就会被再一次访问。
  • 空间局部性：如果一个数据被访问过，那么他周边的数据也会被访问。

所以一个网站中，20% 的数据要承载 80% 的访问量，遵循著名的 “2:8法则”。

使用 Nginx 的缓存时需要注意：

  1. 先定义后再使用。
  2. 不能交叉使用，http 要用 http 专有的缓存插件；fastcgi 要用 fastcgi 专有的缓存插件，不同的协议不能混用。
  3. 缓存有淘汰机制 LRU，即选择最久未使用的数据予以淘汰。
  4. 缓存数据通过"键值"格式存储，为了查找时加速，其中“键”是将请求的 url 做 HASH 计算后，按照 HASH 值中字母或数字的顺序分级存放，Nginx 中最多 3 级。

下面列举一些常用的 HTTP 缓存模块的参数。

ngx_http_proxy_module 代理模块的缓存功能¶

ngx_http_proxy_module 模块除了能配置 proxy 参数，还能配置 缓存参数。

proxy_cache_path¶

定义可用于 proxy 功能的缓存，最多 3 级，定义后，需要引用才能使用。

可用位置：

  • 定义时必须在 http；
  • 引用时位置 server, location。

示例：

  • 定义：
    • proxy_cache_path /var/cache/nginx levels=1:1:2 keys_zone=webcache:10m;
  • 引用：
    • proxy_cache webcache;

proxy_cache¶

指明要调用的缓存，或关闭缓存机制。

  • proxy_cache zone | off;
  • 可用位置：http, server, location

proxy_cache_key¶

  • 定义缓存中用什么数据当做“键”。
  • proxy_cache_key string;
  • 默认值：proxy_cache_key $scheme$proxy_host$request_uri;
  • 生产环境中可能不同的 url 或 uri 对应的是相同的资源，为了提高缓存的命中率，需妥善定义“键”。

proxy_cache_valid¶

  • 定义对特定响应码的响应内容的缓存时长。
  • proxy_cache_valid [code ...] time;
  • 可以多次定义不同响应吗的缓存时长。

proxy_cache_use_stale¶

  • 当无法从原始服务器获取最新的内容时，Nginx 可发送在其缓存中的陈旧的文件，而不是向客户端传达错误信息。
  • proxy_cache_use_stale error | timeout | invalid_header | updating | http_500 | http_502 | http_503 | http_504 | http_403 | http_404 | off ...;

proxy_cache_methods¶

  • 客户端使用此指令中指定的请求方法时，使用缓存响应。 GET 和 HEAD 方法虽然是默认的，但还是建议明确指定它们。
  • proxy_cache_methods GET | HEAD | POST ...;
  • 可用位置：http, server, location

四、FastCGI 代理模块¶

在中小型站点中，使用 php 是最合适的选择，那么在搭建 LAMP 时，php 有两种部署方式：

1、作为 httpd 的模块运行，缺点是 httpd 要处理静态和动态的请求，流量大时 web 服务器的压力较大。

  • 当 httpd 是 prefork 时，是 php_module
  • 当 httpd 是 worker 时，是 zts_module

2、通过 FastCGI 的方式运行，优点是动静分离。

  • fpm：fastcgi process manager

第 2 中方式中的 fpm，其实就是个 FastCGI 进程管理器，它在 php 解释器的基础上，加了一个壳，使 php 可以监听在一个套接字上，通过 FastCGI 接收并处理并发请求。

而 FastCGI 为了处理并发请求，使用的 MPM（多路工作模块）和 httpd 中的 frefork 一致，采用 M/W 的方式，Master 进程管理 Worker 线程，Worker 线程处理 php 请求。

在 Nginx 中，暂时并不支持像第 1 种那样以模块的方式运行 php，只能通过调用 Nginx 自己的 FastCGI 模块，来和后端的 fpm 通信。

ngx_http_fastcgi_module 模块¶

使用这个模块将请求通过 FastCGI 传递给后端服务器，下面列出几个常用配置参数。

fastcgi_pass¶

  • fastcgi_pass ADDRESS;
  • ADDRESS 为 fastcgi server 的地址。
  • 可用位置: location, if in location。
  • 注意：不需要加任何协议开头，如：http://, fastcgi:// 等。

fastcgi_index¶

  • fastcgi_index index.php ;
  • 设置 fastcgi 默认的主页资源。

fastcgi_param （传递参数）¶

  • fastcgi_param parameter value [if_not_empty];
  • 向后端 FastCGI 服务器传递参数，和静态资源不同，动态资源需要很多客户端请求时的参数。

假设 nginx 把请求代理至后端 fpm 时，fpm 是没办法直接使用 nginx 中客户端连接信息的参数的，因为 nginx 中变量的表达方式，和 fpm 中变量的表达方式不一样，例如 nginx 中的变量都是小写，fpm 却是大写等等，所以 nginx 代理 fpm 时，要把自己的变量名转成 fpm 的变量名才行。

fpm 中需要传递的参数很多，nginx 安装好后，在 /etc/nginx/ 目录下会自动生成一个 fastcgi.conf 文件，该文件中包含了使用 fpm 时 必备的参数。
