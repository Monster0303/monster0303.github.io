一、Ingress 是什么？¶

在使用 Service 时，无论是 iptables 还是 ipvs 都是四层调度，这就代表它们都无法作为 ssl 会话卸载器使用，这就要引入另一个资源来使用 7 层的调度。

Ingress 可以想象成是一种七层的路由规则，对于 http 和 https 的流量，能够把来自客户端的对某个 URL 的访问，转换为对内部某个特定 Pod 的访问。

Ingress 能够让你用一种 通用配置格式 来配置规则，这个语法不是 Nginx 格式，也不是 Apache 格式。但是写好通用配置后，它能够自动转为 Nginx 或 Apache 等程序的配置格式。所以你只需要告诉它，需要把哪个 Url 代理到哪个 Url 就完了，至于怎么转换，那是自动进行的。

负责 入站，在 7 层把外部的访问代理到 Pod 中。

Ingress Controller 是什么？¶

Ingress 只是一种规则，而这个规则需要一个实实在在存在的进程，这个进程能够理解 Ingress 定义的配置信息（规则），并将 Ingress 规则转为 Nginx、HAProxy 等程序能够识别的配置文件，这就是 Ingress Controller。

注意：

  • Ingress Controller 并不是所谓的控制器，它只是一个 Pod，被称为 Ingress Controller 而已。
  • 这个 Pod 的作用是专门用来确保 Ingress 中定义的配置，能够实时加载并生效的。他与后端 Pod 是一对多的，后边可能有无数个 Pod。
  • 这个控制器没有被打包进 Controller-manager 中统一管理，它既可以运行为一个系统级的守护进程，也可以运行在 Pod 中。但是因为它很复杂，还经常需要变动，为了便于配置，所以不得不将它运行为 Pod。
  • Ingress Controller 也是 Pod，它也应该有自己的 Pod Controller，或许使用 DeployMent 类型来维持多个副本，或许使用 DaemonSet 类型，在每个节点上都运行一份。

综上所述，如果要在 k8s 集群中使用 ssl 会话卸载器之类的 7 层代理功能，就需要使用以下两个组件，而不是用 Service。

  • Ingress Controller：用来把客户端的访问代理到其他的 Pod 上；
  • Ingress：用来定义代理规则。

但是 Ingress 并不支持使用标签选择器来挑选后端的 Pod，所以当后端有多个 Pod 时，它需要由 Service 辅助完成此类功能。在这里 Service 还有另一个功能，就是当 Pod 发生变化时，比如数量增加了一个，也需要通过 Service 来 帮助 Ingress 感知的。

  • 注意：这时 Service 仅仅为 Ingress 提供标签选择器的功能，并不代理网络流量，它不是 Ingress 真正的后端，客户端的访问不会到达 Service，而是从 Ingress 直接到达 Pod。

二、Ingress 在生产环境中的使用方法¶

方案一¶

这是标准方法之一。

假设目前有多个后端 Pod 需要 7 层的代理服务，这时启动一个 Ingress Controller 来代理对后端 Pod 的访问。

这个 Ingress Controller 也是一个 Pod，为这个 Pod 添加一个 NodePort 类型的 Service，这样任何一台 Node 都可以作为引入外部流量的入口。

但是缺点是 NodePort 一般情况下使用的不是标准端口，于是就需要在节点外再加一个代理服务器，来将非标准端口转换成 80、443 之类的标准端口。

大致流程如下：

客户端 -> 代理节点（监听 80 标准端口） -> Node 节点 -> Service(Nodeport 监听着非标准端口) -> Ingress Controller 的 Pod -> 后端 Pod

这个流程太繁琐了，一点都不让人省心，所以为了简化这个过程，就有了下面的方案二

方案二¶

在集群中准备几个特殊的 node，这几个 node 不仅有私网地址，还有公网地址。

然后为这几个节点打上污点或使用标签（ds 支持节点标签选择器），通过 ds 控制器让 Ingress Controller 的 Pod 只在这几个有公网地址的节点上运行，并为 Pod 添加一个 hostPort 类型的 Service，直接映射到物理节点的某个端口。

如果为这几个 Node 节点预留了 80 或 443 这样的标准端口，这样当客户端访问这几个特殊节点中某个节点上的某个公网端口时，就等于访问当前节点中的 Ingress Controller 的端口，而后 Ingress Controller 就可以把访问代理至后端的 Pod 上了。

大致流程如下：

客户端 -> Node 节点 -> Service（hostPort） -> Ingress Controller 的 Pod -> 后端 Pod

方案三¶

还有一个更简化的方案。

在方案二的基础上，不使用 hostPort，而是让 Pod 共享节点的网络名称空间（hostNetwork）。

实现了无需代理，一次直达 Pod。

大致流程如下：

客户端 -> Node 节点（hostNetwork） == 访问 Ingress Controller 的 Pod -> 后端 Pod

Ingress 版 Nginx¶

现在有一个问题摆在眼前，现有的 Nginx、Haproxy 等都不是云原生的程序，所以并不知道如何从 API Server 中读取 Ingress 的规则，也无法看懂 Ingress 中定义的规则。这可怎么办呢？

好在 k8s 官方把 Nginx 改造，变成一个自带 Ingress Controller 功能的 Nginx。所以要在 k8s 中使用的话，需要使用经过改造的 Nginx。

定义 Ingress 规则¶

Ingress 规则本身也是 k8s 中的一种资源，它的 API 版本是 extensions/v1beta1。

注意：一个集群中可能运行着不止一个 Ingress Controller，比如同时部署了 Ingress-Nginx 和 Ingress-HAProxy，那么该由哪个来转换规则呢？

Ingress 支持在 annotations 中定义一些规则，那么在这里就可以定义由谁来转换的规则：

  • kubernetes.io/ingress.class: "nginx"：让 Ingress-Nginx 来转换；
  • kubernetes.io/ingress.class: "haproxy"：让 Ingress-HAProxy 来转换。

Ingress-Nginx 其实还支持很多转换规则，可以定义的很细节，在这里可以查看全部：

  • https://github.com/kubernetes/ingress-nginx/blob/master/docs/user-guide/nginx-configuration/annotations.md
