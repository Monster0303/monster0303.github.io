一、基础概念¶

存储设备的类型¶

  • DAS（直接附加存储）：IDE、SATA、SCSI、SAS、USB；
  • NAS（网络附加存储）：NFS、CIFS；
  • SAN（存储区域局域网络）：也是网络存储，只不过提供给客户端的存储接口是块设备接口。
    • SCSI 协议只能使用 SCSI 专用线来传输数据，但是因为 SCSI 协议是分层的，所以有人就把底层的物理传输接口改换为光纤和以太网，就有了 FC SAN、iSCSI。

Ceph 的基础概念¶

Ceph 是什么？¶

Ceph 是一个对象（object）式存储系统，它把每一个待管理的数据流（例如一个文件） 切分为一到多个固定大小的对象数据，并以其为原子单元完成数据存取

  • 对象数据的底层存储服务是由多个主机（host）组成的存储集群，该集群也被称之为 RADOS（Reliable Automatic Distributed Object Store）存储集群，即可靠、自动化、 分布式对象存储系统；
  • 底层存储服务由 librados 接口向外提供，这个接口是 RADOS 存储集群的 API，它支持 C、C++、Java、Python、Ruby 和 PHP 等编程 语言。

存储数据时，可以自己编写一个客户端软件，去调用 librados 的 api 接口。而 ceph 为没有编程能力的用户在 librados api 之上提供了 2 个 api 接口，和一个不依赖 librados api 的接口：

  • RBD：基于内核模块 librbd 给系统作为块设备使用，或者直接给 qemu 作为虚拟机的块设备使用；
  • RADOSGW：通过 http、https 协议提供对象存储。
  • CEPH FS：这个接口不依赖于 librados，是 RADOS 集群的一部分，提供文件存储功能，J 版后才生产可用，可以直接挂载；

Ceph 集群的主要组件¶

  • osd：通常情况下，主机上的每一块磁盘设备被称为一个 osd（Object Storage Device），而这个主机被称为 host，由多个 host 组成的集群被称之为 RADOS Cluster。在集群中，还有其他的节点，比如元数据服务器（mon）和 mgr；
  • mon：用来记录集群有多少个 host，每个 host 中有多少个 osd，每个 osd 是否健康等等的信息，持有整个集群的运行状态图。所以 mon 节点是用来存放集群元数据，而非文件元数据，也称之为监视器。因此需要做高可用，内部使用 Paxos 协议实现；
  • mgr：在 L 版之后引入了 mgr，负载监控 Ceph 集群。之前是由 mon 负责。

CRUSH 算法存数据的步骤¶

通过某一类客户端接入，接入的时候借助 ceph 的 api 接口，把一个文件切割成固定大小的对象，把对象的名字做 Hash，然后对所在 Pool 中的 PG 总数做一致性 Hash 计算，结果会落在某一个 PG 上。

PG 和 Pool 是逻辑概念，不存在实体，每一个 Pool 叫一个存储池，存储池的容量大小取决于底层的存储空间，而数据最终放在哪个 osd 上，是靠 CRUSH 完成的。

PG 根据副本数量和存储池的类型，找到足量的 osd 来存储，一个 PG 内的所有对象，都放在同一个 osd 中，数据第一次先写到主 osd 中，再由主 osd 同步给其他的从 osd。

推荐阅读：https://www.liukui.tech/2019/05/10/存储基础和Ceph/

Monitors 的主要作用¶

mon 是一个运行在主机上的守护进程，被称为 ceph-mon，维护着集群运行图（Cluster Map），包括 5 个集群运行图：

  • monitor map
  • manager map
  • OSD map
  • CRUSH map
  • PG map

mon 节点还负责维护认证信息，并通过 CephX 协议进行认证。如果客户端规模大，mon 节点可能会成为瓶颈，不过好在 mon 是无状态的，可以部署多个，一般是 3 个起的奇数个。

Managers 的主要作用¶

manager 也是一个运行在主机上的守护进程，被称为 ceph-mgr，应该有两个或以上数量的节点。跟踪运行时的指标数据和 Ceph 集群的运行状态，包括存储空间利用率、性能指标、节点的负载等等。

Ceph OSDs 的主要作用¶

每个 OSD 都会有对应的守护进程，专用于管理一个 OSD。OSD 用于存储数据，负责数据的复制、恢复、平衡，给 Monitors 和 Managers 提供监视信息。

MDSs 的主要作用¶

Metadata Server 的作用是当使用 Ceph Filesystem 时，用来存储文件的元数据，不是必要组件。

当面向客户端提供服务时，CephFS 和 RADOSFS 都需要单独运行一个守护进程，而 ODS 则不需要，它是依靠在 libredos 上的一个模块来提供服务。

管理节点（Admin Host）¶

Ceph 的常用管理接口是一组命令行工具程序，例如 rados、ceph、rbd 等命令，管理员可以从某个特定的 Mon 节点执行管理操作，但也有人更倾向于使用专用的管理节点；

事实上，专用的管理节点有助于在 Ceph 相关的程序升级或硬件维护期间为管理员提供一个完整的、独立的并隔离于存储集群之外的操作环境，从而避免因重启或意外中断而导致维护操作异常中断。

6. 从 RADOS 集群移除 OSD¶

Ceph 集群中的一个 OSD 通常对应于一个设备，且运行于专用的守护进程。在某 OSD 设备出现故障，或管理员出于管理之需确实要移除特定的 OSD 设备时，需要先停止相关的守护进程，而后再进行移除操作。对于 Luminous 版及其之后的版本来说，停止和移除的命令格式分别如下所示:

  1. 停用设备：ceph osd out {osd-num}
  2. 停止进程：sudo systemctl stop ceph-osd@{osd-num}
  3. 移除设备：ceph osd purge {id} --yes-i-really-mean-it

若在 ceph.conf 配置文件中，静态定义了类似如下的 OSD 配置信息时，管理员则需要在删除 OSD 之后手动将其删除。

而对于 Luminous 之前的版本，管理员需要依次手动执行如下步骤删除 OSD 设备:

  1. 于 CRUSH 运行图中移除设备：ceph osd crush remove {name}
  2. 移除 OSD 的认证：key:ceph auth del osd.{osd-num}
  3. 最后移除 OSD 设备：ceph osd rm {osd-num}

三、开启 Ceph 存储集群的访问接口¶

使用块设备接口 RBD¶

Ceph 块设备，也称为 RADOS 块设备（简称 RBD），是一种基于 RADOS 存储系统支持超配（thin-provisioned）、可伸缩的条带化（Object）的数据存储系统，它通过 librbd 库与 OSD 进行交互。

RBD 为 KVM 等虚拟化技术和云 OS（如 OpenStack 和 CloudStack）提供高性能和无限可扩展性的存储后端。这些系统依赖于 libvirt 和 QEMU 程序，与 RBD 进行集成。

客户端基于 librbd 库即可将 RADOS 存储集群用作块设备，不过，用于 rbd 的存储池需要事先启用 rbd 功能并进行初始化。
