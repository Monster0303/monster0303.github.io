一、Dockerfile 中需要了解的一些东西¶

制作 Docker 镜像可以使用 docker commit 命令，需要先启动镜像，然后打包时，会把最上层可写层的数据，以及依赖的关系，完整保存在联合文件系统中。但其实这整个过程，可以通过另一种方式完成，就是利用制作镜像的相关源码中的指令来自定义镜像，而不用自己手工完成。

安装软件包、创建文件夹、从本地指定或者互联网复制文件等，这就是 Dockerfile，它会自动创建一个镜像，把之前需要手动操作的步骤，隐式完成，并打包成镜像。

1. Dockerfile 的一些特性¶

  • Dockerfile 其实就是构建 Docker 镜像的源码。
  • Dockerfile 能够通过读取 file 文件里的指令 (instruction) 来构建一个镜像。
  • Dockerfile 其实仅仅是个纯文本文件，只不过把文件里的内容作为 Dockerfile 的指令 (instruction) 来运行。
  • 这个指令 (instruction) 是有特定格式的，只能使用语法格式当中定义的，能被 docker 识别的指令。

2. Dockerfile 的工作环境¶

Dockerfile 在工作时，需要一个工作目录，我们称之为 工作车间。

工作车间相当于是宿主机文件系统上的一个目录，这个目录内只放置构建镜像需要使用的 Dockerfile 文件，和打算复制的文件。

这个目录其实可以理解成 Docker 的存储卷，构建镜像时，从这个卷里找到文件，并复制到容器，所以需要注意的是，如果文件放在其他位置的话会无法复制，所以要复制的文件必须放在当前文件夹内。

同时，Dockerfile 文件本身也只能叫 Dockerfile。

3. 构建镜像¶

需要注意的一些问题：

  • 指令本身并不区分大小写，但是按照约定俗成的惯例，指令一般写成纯大写。
  • 没有多余的语法，所有指令按顺序执行。
  • 一般来讲，第一行是 FROM，但是对于新版的 file，这个不是强制要求的了。
  • 和手动构建镜像的区别是：每一条指令都会生成一个镜像层。
  • 最好把关系联系紧密的操作放在同一层。

关于镜像层的数量控制：

分类 优点 缺点
镜像层多 可以控制的很精细，分发给其他容器时容易共享 第一次写入是 cow 机制，对性能略有影响
镜像层少 镜像层比较整体化，性能略优 不便于自由分发

4. 启动脚本 entrypoint.sh¶

大部分程序的定制是通过配置文件来实现的，而镜像中程序的配置文件已经写死在镜像中了，这就导致了如果有多个部署环境，需要为每个环境下的每个程序制作各不相同的 Docker 镜像，这就很头大了。

所以现在面向云原生开发的应用程序都支持 --> 容器启动时直接读取 Docker 宿主机的环境变量，然后根据环境变量把参数传入到配置文件内。

但是一些老牌的程序，并不支持从环境变量中传入配置，为了解决这个问题，便引入了指定环境变量的方法，如果程序不支持直接获取宿主机环境变量，比如 mysql 的镜像启动时，就需要使用 -e MYSQL_ROOT_PASSWORD 的选项来指定 root 密码，这就解决了老的程序不支持读取宿主机环境变量的问题，就像有句名言是这么说的 计算机科学领域中任何两个不衔接的层，都可以通过增加一个间接的中间层来解决，这个脚本就是衔接新老程序的中间层。但是还有一个问题，如果参数太多了，就需要在 shell 中输入很长的参数，这也很头大。

所以 Docker 支持在启动时使用脚本，作为容器启动的第一个进程，脚本运行完后，就把原本要运行的程序启动，然后自动退出，这个脚本就叫 entrypoint.sh，文件本身也叫这个。

5. 设置忽略的文件¶

如有在 Dockerfile 的工作车间目录下，有不希望在构建镜像时复制的文件，可以通过 .dockerignore 文件配置，Docker 会自动忽略这个文件以及文件内所有指定的文件。

操作准则基本同 COPY 指令，但需要注意的是：

8. ENV¶

用于为镜像定义所需的环境变量，并可被 Dockerfile 文件中位于其后的其它指令（如 ENV、ADD、COPY 等）所调用，调用格式为 $variable_name 或 ${variable_name}。

当运行容器时，环境变量将保持不变。可以使用 docker inspect 查看值，如果对变量进行更改请使用 docker run --env|-e <key>=<value>。

语法：

需要注意的是：

  • 第一种格式中，<key> 之后的所有内容均会被视作其 <value> 的组成部分，因此，一次只能设置一个变量。
  • 第二种格式可用一次设置多个变量，每个变量为一个 <key>=<value> 的键值对，如果 <value> 中包含空格，可以以反斜线 \ 进行转义，也可通过对 <value> 加引号进行标识。另外，反斜线也可用于续行。
  • 定义多个变量时，建议使用第二种方式，以便在同一层中完成所有功能。

注意：ENV 定义的变量，会在容器启动后传入到容器内。

注意：使用 ENV 指令定义的环境变量始终覆盖同名的 ARG 指令。

9. ARG¶

ARG 是 Docker 新版本中支持的参数，能够在 build 阶段使用 --build-arg key="value" 定义一个变量，替换 Dockerfile 中 ARG。

注意：ARG 定义的变量，容器启动后不会传到容器内。

注意：建议在 build 阶段使用 ARG，比如在 Dockerfile 中定义了 ARG，那么 build 阶段就可以使用 --build-arg key="value" 进行指定了，由于容器启动后不会传到容器内，那么只有在 build 阶段的指令 COPY、ADD 等会受益。而 ENV 则建议在 RUN 阶段使用，由于 ENV 可以传入到容器内，那么 build 和 run 阶段的所有指令，包括 CMD、ENTYRPOINT 等都将受益，启动容器时就可以通过 --env 或 -e 来指定新变量，传入到容器内。

注意：使用 ENV 指令定义的环境变量始终覆盖同名的 ARG 指令。

ARG 的构建参数和 ENV 的效果一样，都是设置环境变量。所不同的是，ARG 所设置的构建环境的环境变量，在将来容器运行时是不会存在这些环境变量的。但是不要因此就使用 ARG 保存密码之类的信息，因为 docker history 还是可以看到所有值的。

14. HEALTHCHECK¶

HEALTHCHECK 指令告诉 Docker 应该如何进行判断容器的状态是否正常。

在没有 HEALTHCHECK 指令前，Docker 引擎只可以通过容器内主进程是否退出来判断容器是否状态异常。很多情况下这没问题，但是如果程序进入死锁状态，或者死循环状态，应用进程并不退出，但是该容器已经无法提供服务了。在 1.12 以前，Docker 不会检测到容器的这种状态，从而不会重新调度，导致可能会有部分容器已经无法提供服务了却还在接受用户请求。

而自 1.12 之后，Docker 提供了 HEALTHCHECK 指令，通过该指令指定一行命令，用这行命令来判断容器主进程的服务状态是否还正常，从而比较真实的反应容器实际状态。

当在一个镜像指定了 HEALTHCHECK 指令后，用其启动容器，初始状态会为 starting，在 HEALTHCHECK 指令检查成功后变为 healthy，如果连续一定次数失败，则会变为 unhealthy。

语法：

HEALTHCHECK 支持下列选项：

  • --interval=<间隔>：两次健康检查的间隔，默认为 30 秒；
  • --timeout=<时长>：健康检查命令运行超时时间，如果超过这个时间，本次健康检查就被视为失败，默认 30 秒；
  • --retries=<次数>：当连续失败指定次数后，则将容器状态视为 unhealthy，默认 3 次。

和 CMD, ENTRYPOINT 一样，HEALTHCHECK 只可以出现一次，如果写了多个，只有最后一个生效。

在 HEALTHCHECK [选项] CMD 后面的命令，格式和 ENTRYPOINT 一样，分为 shell 格式，和 exec 格式。

  • 命令的返回值决定了该次健康检查的成功与否：

    • 0：成功；
    • 1：失败；
    • 2：保留，不要使用这个值。
